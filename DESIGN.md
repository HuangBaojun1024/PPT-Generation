# DESIGN.md — 决策日志

> 输入一个 JSON（topic / brief / audience），输出一套 25–30 页、风格统一、有叙事线的 `.pptx`。
> 考核维度：美观度 / LLM 成本 / 生成速度，交付 `beauty-max`（拉满美观）与 `balanced`（trade-off）两版。
> 完整技术方案见 `docs/pipeline-proposal.md`，本文只讲「决策」。

---

## 1. 架构图 + 数据流

核心判断：**不让 LLM 直接吐 pptx 坐标，而是让它写 HTML/CSS**——LLM 的前端排版能力远强于摆 pptx 坐标，且 HTML 能被浏览器精确测量，天然自带质检能力。pptx 只是最后一道「格式转换」。

```
输入 JSON { topic, brief, audience }  +  --profile  --preset
        │
 ① 规划 planner (gpt-5.5 high · 1 次)      ← 唯一的"贵"调用
        │  DeckPlan：叙事弧 / 章节 / 色板+双字体 / 选中模板
 ② 分镜 writer (gpt-5.5 low · 按章并行)
        │  SlidePlan[]：每页 pageType / 文案 / 素材需求
 ③ 策略 choose-strategy (纯代码 · 0 LLM)   ← 免费的确定性规则
        │  RenderDecision[]：版式变体 + 素材清单 + 生图预算分配
        ├─ ③.5 素材并行预取：GPT-Image-2 生图 / Phosphor 图标 / SVG
 ④ 渲染 coder (逐页并行 → 单页 HTML)
        │  共享同一份 tokens.css（构造性一致）
 ⑤ 评估 evaluate
        │  L1 确定性检查 (Playwright DOM 测量, 0 LLM) → 单页修复循环
        │  L2 reviewer 拼图审查 (opus-4.7 thinking, 可关) → 单页返工
 ⑥ 合并 merge-pptx (纯代码)
        │  每页 HTML → 混合转换 → pptx 页
   final.pptx + metrics.json + audit.json + html/
```

**LLM 只花在 3 个环节**：① 规划 1 次、② 分镜约 5–7 次（并行）、④ 渲染 25–30 次（并行）+ 返工页。③ 策略和 ⑥ 合并是零 LLM 的纯代码。设计原则一句话：**贵的推理只花在 1 次全局规划上，其余都是可并行的便宜调用或零成本代码。**

`--profile`（视觉预算）与 `--preset`（各角色用什么模型）是**两个正交开关**，可自由组合。

## 2. 模型选型

代码里**只认「角色」不认模型名**（`getModel("coder")`），模型/代理商/密钥全部外置到 `config/models.json` + `.env`，换供应商是改配置不是改代码。四个可独立配置的角色 + 生图：

| 角色 | 拉满档（full preset） | 为什么是它 | 比较过的替代 |
|---|---|---|---|
| planner 规划 | gpt-5.5 (reasoning high) | 全局叙事+风格质量由它兜底，只调 1 次，值得上高推理 | 试过 low 推理，叙事弧明显更平 |
| writer 分镜 | gpt-5.5 (low) | 结构已被 zod schema 约束，低推理并行即可 | glm-5.1 作降级线 |
| coder 写 HTML | claude-fable-5 (haoai) | 写前端代码能力最强，页面美观度上限由它决定 | gpt-5.5 作降级线；实测 gpt-5.5 也能出合格页，故 test preset 直接用它 |
| reviewer 审查 | claude-opus-4-7 (extended thinking) | 一致性判断是跨页比对推理，需要视觉+思考；低推理多模态判不稳 | gpt-5.5 做不了可靠一致性审查 → 干脆做成开关 |
| image 生图 | gpt-image-2 (saturday → apimart 兜底) | 已有链路、同步接口快 | — |

**本次 5 套 demo 全部用 `test` preset**：planner/writer/coder 都是 gpt-5.5，reviewer 关闭（只留 L1 确定性检查）。选 gpt-5.5 很大一部分原因是**性价比**：它在各大中转站的价格都打得比较低（本项目走的网关计价 $1.25/M 输入、$10/M 输出，见 §5 单价表），在能力够用的前提下是最便宜的主力选择。实测它在构造层约束（共享 tokens + LayoutMap + L1 修复循环）下已能稳定出合格页，一致性主要靠「构造」而非「审查」兜（见 §3），因此在成本/速度维度下 reviewer 的边际收益不足以支撑 opus-4.7 的价格。full preset 保留给「美观度还要再往上顶」的场景。

## 3. 风格一致性怎么保的（核心难点）

策略：**构造性保证优先（编译出来就是一致的，约 90%），评估只兜底漏网的 10%。** 一致性不是「每页各自发挥后再去检查」，而是「从一个源头编译出来」。

构造层（零成本、确定性）：

1. **单一 Design Token 源**：DeckPlan 的色板/双字体编译成唯一一份 `tokens.css`，**全套 25–30 页共享**。主色/辅助色/背景/文字色、字号阶梯、间距、圆角、安全边距全走 CSS 变量。改一处，全套变。
2. **图片统一色调滤镜**：所有 GPT-Image-2 生成图在 HTML 层统一叠主色 duotone/tint 色罩，把「风格各异的生成图」强行拉进同一色系——这是消除 fullImage 页与模板页视觉断层最便宜的手段。
3. **Style Anchor 生图锚**：每条生图 prompt 都拼上固定的「风格锚文案 = 风格描述 + 色板 hex + 负面清单(no text/watermark/logo)」，统一模型/宽高比/seed 策略，保证同一套视觉语言。
4. **图标硬约束**：全套只用 Phosphor **一套库 + 一个字重**（beauty-max=duotone / balanced=regular），SVG 内联、`currentColor` 继承主色 token，天然与配色一致。全套 ≤4 色、≤2 字族。

评估层（抓漏网的 10%）：L1 DOM 测量查对比度/最小字号/安全边距/文字溢出/装饰压字；L2 reviewer 一次看 30 页缩略图专找配色/密度跳戏的离群页（本次关闭）。

## 4. 多样性怎么保的

一致性锁的是「视觉**系统**」（色板、字体、图标库、网格、留白节奏）；多样性放开的是「页面**变体**」。三个层级制造差异，同时不破坏系统：

- **模板选择**：3 套 LayoutMap 模板（aurora-consulting 咨询报告风 / matcha-editorial 杂志编排风 / midnight-technical 深色技术风），planner 按 topic+audience 选最合适的一套，不同输入进不同模板。
- **叙事结构**：PPT 的章节目录和页面顺序不是写死的套路，是 planner 拿到主题后现场设计的。同样一条 pipeline，「年度复盘」会长成"时间线 + 得失盘点"，「说服 CEO 用 Rust」会长成"现状痛点 → 方案对比 → 收益与计划"，「京都两日游」会长成"行程场景 + 预算清单"。目录结构本身就不同，每页用什么页型（时间线页/对比页/清单页）自然也不同。
- **同模板内变体轮换**：同一 pageType 有多个 variant，配色在主色域内派生，封面/章节页生图各不相同。

产出是「同系不同款」，不是「同一个模板填空」。这也正是 `--profile` 之外，5 套 demo 彼此差异明显的原因（见各 demo 的 `final.pptx`）。

## 5. 成本与时延实测（5 套 demo × 两版）

test preset（全 gpt-5.5，reviewer 关闭）。约束线：< $10 / 套、< 30min / 套。数据取自各套 `metrics.json`。

**成本统计口径**：代码内建埋点（`src/metrics.ts`），每次 LLM 调用记录实际 tokens 用量，按 `config/pricing.json` 的单价折算：文本成本 = 输入 tokens × 输入单价 + 输出 tokens × 输出单价，图片按张计。**单价用的是中转站（网关）的计价，不是模型官方牌价**——项目走 saturday/haoai/apimart 中转，官方价折算会失真，所以单价表独立成配置、按中转站口径填：

**读数**：
- **两版的成本差几乎全在生图**：文本成本两档都在 ~$1（gpt-5.5 逐页写 HTML），图片从 balanced 的 $0.35（5 张 1k）涨到 beauty-max 的 $2.28（12 张 2k）。所以 beauty-max 的成本大头是生图，而 balanced 反而是文本占多数——**要压成本先砍生图张数**，这正是两档位的杠杆。
- 两版共用同一条 pipeline 和同一套模板，差异只在「花多少钱买视觉」：beauty-max 把预算砸在生图+精修，balanced 用更多确定性渲染（图标/CSS 装饰）换成本与速度。
- 全部远低于 $10 / 30min 上限。

## 6. 踩坑和取舍（试过被否 / 被迫改路线）

| 方案/坑 | 结论 | 原因 |
|---|---|---|
| 开发中途 Claude Code 账号被封 | 换途径解决 | 先尝试改用 GPT-5.5 顶替做编码助手，但 5.5 被降智、整体效果较差；最后重新购买 Fable-5 的 API 用量才恢复开发节奏 |
| 不用 HTML 中转、让 LLM 直接产 pptx（想简化架构） | **试过，被否** | 排版效果比 HTML 差一截（LLM 摆 pptx 坐标的能力远弱于写 CSS），且产物无法被浏览器测量质检，等于放弃了整个 L1 检查层 |
| 整页截图转 pptx（最简单） | 部分保留为**兜底** | 文字不可编辑、不清晰；改为「背景光栅化 + 行级文字覆盖」混合模式，对不齐才退化整页截图 |
| 逐页 reviewer 审查 | 否 | 30 页 = 30 倍成本；改为拼图审 1 次即可抓「不一致」（不一致本质是跨页对比） |
| pptx 原生 chart | 否 | 样式丑、不可控；图表走确定性 SVG 渲染 |
| 官方模型定价直接入账 | 改为**配置化** | 网关计价与官方价不一致，`config/pricing.json` 按实测账单校准 |
| about:blank 里加载 file:// 子资源 | 坑 | Chromium 安全策略拦截 → 改为 HTML 落盘后 `goto file://` |
| tsx(esbuild keepNames) 注入 `__name` | 坑 | 污染页内 JS → `addInitScript` 注入 shim |
| 并行任务共用 temp 目录互删 | 坑 | 改按 PID 隔离 `pptgen-${pid}`，本次 10 套并行才能安全跑 |
| 本机无 LibreOffice | 取舍 | 无法 headless 转 png 验收 → 用 pptx XML 抽查（大字号计数）+ Keynote 人工开 + HTML→png 预览三管齐下 |

## 7. AI 协作复盘

**关键方案是我定的，AI 负责填充与实现**
- **HTML+CSS 作中间产物**和 **LayoutMap 模板**（用 Map 描述页型→布局区块的约束）这两个决定美观度上限与一致性根基的方案是我提的。
- **主流程**（规划→分镜→策略→渲染→评估→合并的六层结构）以及**多级检查机制和检查后的流转**（不过关的页按问题分类：样式问题回渲染层重写 HTML、素材问题回策略层重定方案，单页级返工不重跑全流程）也是我给的思路。
- 风格一致性上，「**全套统一样式**」（后来落地为单一 tokens.css）和「**生图注入风格统一的提示词**」（后来落地为 Style Anchor）是我想的。

**AI 提的、我照做了**
- 用 **Playwright** 一个引擎承担渲染/测量/截图/质检的闭环——这是 AI 在「HTML 中转」方案之上填充的关键实现细节，也是 L1 检查能做到零成本的原因。
- **行级文字覆盖**：用 Range API 逐字符取字形矩形、按行分组、每行一个 pptx 文本框——比我最初想的「整段一个框」对齐精度高得多。
- 一致性构造层的补充手段：图片统一色调滤镜、Phosphor 单库单字重硬约束，以及「构造性保证优先、评估只兜底」的分层表述。

**AI 提的、我推翻了**
- AI 一开始倾向「reviewer 常开、逐页精修」。我推翻：成本维度下这不划算，改成 reviewer 可开关、且用**拼图审**而非逐页审；本次 demo 直接关掉 reviewer，用零成本的 L1 + 构造层约束扛一致性。事实证明够用。
- AI 建议直接用模型官方定价折算成本。我推翻：走的是内部网关，计价口径不同，改成 `pricing.json` 配置化、按实测校准，否则 metrics 里的成本数是假的。
- AI 想在 planner 也用低推理省钱。我推翻：planner 只调 1 次、是叙事质量的总闸门，这一次值得上 high 推理；要省该省的是生图张数。

**"AI 跑偏、我把它拽回来"的具体例子**
- 例 1：slide-10 时间线页，coder（AI）反复用绝对定位/负偏移把节点圆圈和图标怼进相邻文字列，视觉上图标压字。AI 的自然倾向是「再 re-prompt 一次让它挪一下」，但这是碰运气。我把它从「反复试」拽到「约束 + 可验证」：给 coder 加死规则（禁止悬挂定位侵入相邻列、兄弟行结构必须一致、图标不够就整行改用色点/编号），**同时**在 L1 加了确定性的「装饰元素×文字重叠」检测（扫小型可见装饰元素与文字矩形求交，命中即进修复循环）。先用出问题的旧页验证检测能精准报出重叠，再全量重跑——42 条修复记录里 0 条装饰压字，说明构造层直接防住了。
- 例 2：`__name` 和 `about:blank` 两个坑，AI 的首版实现「看起来对」但静默失败。我没有接受 AI 的表层修补，而是定位到根因（esbuild keepNames 注入、Chromium file:// 子资源策略）才修——AI 容易在「能跑就行」处停手，根因排查得我来把关。

**我对 AI 输出做的核验/兜底**
- **确定性检查当地面真值**：不信 LLM 说「这页没问题」，一切排版结论走 Playwright DOM 测量（溢出/重叠/对比度/字号/安全边距/装饰压字），机器判定。
- **每个边界加 zod schema**：DeckPlan/SlidePlan/RenderDecision 全部 schema 校验，LLM 输出不合规就重试→降级，坏数据绝不进下一步。
- **pptx 结果 XML 抽查**：无 LibreOffice 的情况下，直接扫 pptx XML 里的字号，验证 >96pt 的「巨型装饰数字翻车」为 0；再配 Keynote 人工开 + HTML→png 预览肉眼确认关键页（时间线、roadmap）。
- **成本口径校准**：不采信官方定价，按内部网关实测校准 `pricing.json`，保证 metrics 的成本数可信。
